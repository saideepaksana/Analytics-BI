1.‚Äã Introduction
1.1‚Äã Purpose
The project ‚ÄúAnalytics BI, Dashboard Creation and Visualization‚Äù is designed to enable users
to generate analytical reports from ingested data and transform them into meaningful charts and
visualizations. The platform provides an interactive dashboard environment where users can
explore insights dynamically. These dashboards can be exported in various file formats or
embedded into other applications using iframe integration, ensuring flexible sharing and
accessibility of insights.
This document is intended to clearly define the features and requirements of this project, serving
as a comprehensive guide for the development team during design and implementation, and as a
software validation reference for prospective clients to ensure the system meets agreed business
objectives and expectations.

1.2‚Äã Problem statement
Build a platform that enables users to ingest data, build dashboards, run queries, and visualize
KPIs. It should offer drag-and-drop visualization builders, reusable widgets, scheduled reports, and
supports data transformations and joins.
Objective: Empower stakeholders to create meaningful dashboards and derive insights from raw
data without heavy engineering.‚Äã
Scope: ETL-like ingestion, simple SQL/visual query builder, charting library, shareable
dashboards, and scheduled exports.

1.3‚Äã Intended Audience
Businesses, researchers, educators, analysts, and individual users.

2.‚Äã Overall Description
‚Äã

This section provides a comprehensive overview of the Analytics BI, Dashboard Creation,
and Visualization Platform. It describes the system‚Äôs operating environment, the expected level of
technical knowledge from users, general constraints affecting system design and implementation,
and the key assumptions and dependencies considered during development.

2.1‚Äã Product Perspective
The Analytics BI, Dashboard Creation, and Visualization Platform is a generalized data
analytics solution designed for a wide range of users, including businesses, researchers,
educators, analysts, and individual users. The platform enables users to ingest data from multiple
sources, perform transformations and joins, run queries through visual or SQL-based builders, and
create interactive dashboards to visualize KPIs and insights.
It provides drag-and-drop visualization tools, reusable widgets, and configurable charts to
make data exploration simple and intuitive, regardless of domain. The system also supports
scheduled reports, threshold-based alerts, and export or embedding options, allowing users to
share insights easily across different platforms. The goal is to empower users from any field to
analyze and present data effectively without requiring advanced technical expertise.
The product is designed as a stand-alone platform that does not depend on external
software for its core functionality. It will run on both UNIX/Linux and Windows-based systems. The
system is intended to be user-friendly, easy to learn, reliable, and stable, ensuring efficient
performance for the defined problem statement.

2.2‚Äã Product Features
The platform provides the following major features:
1.‚Äã Ingest data from multiple sources, perform transformations and joins, and run queries
through visual or SQL-based builders.
2.‚Äã Create interactive dashboards to visualize KPIs and insights using drag-and-drop
visualization tools, reusable widgets, and configurable charts.
3.‚Äã Support for scheduled reports, threshold-based alerts, and export or embedding options to
easily share insights across different platforms.

2.3‚Äã User Classes and Characteristics
1.‚Äã Non-Technical Users (Viewers/General users): Individuals with basic computer literacy
who require intuitive, drag-and-drop tools for general data exploration.
2.‚Äã Technical Users (Editors/Analysts): Data professionals familiar with SQL and data
workflows who build advanced charts and complex analytical queries.

3.2 Data Cleaning (DTS) and Relationship Mapping
3.2.1 Description and Priority
Priority: High.
Immediately following data ingestion, the system automatically runs Data Transformation Services
(DTS) to standardize the data and prepare it for meaningful analysis. This stage serves two
purposes: first, it automatically fixes data errors; second, it acts as a "relationship maker" by
analyzing the dataset to identify and classify fields as either Dimensions (how the data is
categorized, like 'Region' or 'Date') or Measures (the numerical values, like 'Revenue' or 'Units
Sold'). This automation ensures that the subsequent visualization stage can generate accurate
charts without manual setup.

3.2.2 Stimulus/Response Sequences
‚óè‚Äã Data is successfully ingested into the platform's raw storage.
‚óè‚Äã The system automatically triggers the DTS pipeline to cleanse the data and identify data
types.
‚óè‚Äã The system identifies the schema and field types.
‚óè‚Äã The system automatically maps each field to a "Dimension" or "Measure" category and
establishes the relationships needed for charting.
‚óè‚Äã The system encounters unfixable malformed data.
‚óè‚Äã The system routes these specific records to the Dead Letter Queue (DLQ) and notifies the
administrator while allowing the valid, mapped data to proceed.

3.2.3 Functional Requirements
‚óè‚Äã REQ-2.1: The system shall automatically clean and format ingested datasets without
requiring any manual work from the user.
‚óè‚Äã REQ-2.2: The system shall function as a "relationship maker" by automatically classifying
every data field as either a Dimension or a Measure.
‚óè‚Äã REQ-2.3: The system shall automatically apply fix-it rules, such as replacing missing values
(nulls) with zeros, to ensure data is ready for analysis.
‚óè‚Äã REQ-2.4: The system shall automatically alert administrators if it finds "unfixable" data that
needs manual review in the Dead Letter Queue (DLQ).
‚óè‚Äã REQ-2.5: The system shall automatically link related data tables together so that the
Visualization wizard has all the information it needs for complex charts.

3.3 Query Builder (The Data Wizard)
3.3.1 Description and Priority
Priority: High.
The Query Builder is a user-friendly interface that allows users to select exactly what data they
want to analyze without writing any SQL code. It uses the "Dimensions" and "Measures" identified
during the Data Cleaning stage to guide the user. This "Wizard" ensures that users only ask for
data that makes sense, preventing errors before the visualization starts.

3.3.2 Stimulus/Response Sequences
‚óè‚Äã The user selects a dataset to explore.
‚óè‚Äã The system displays two columns: one for "Dimensions" (categories) and one for
"Measures" (numbers).
‚óè‚Äã The user drags a dimension (e.g., 'Product Category') and a measure (e.g., 'Total Profit')
into the query area.
‚óè‚Äã The system builds the query in the background and shows a "Data Preview" table.
‚óè‚Äã The user applies a filter (e.g., "Only show data for 2025").
‚óè‚Äã The system updates the preview to show only the filtered results.

3.3.3 Functional Requirements
‚óè‚Äã REQ-3.1: The system shall provide a Visual Query Builder that allows users to create data
requests by dragging and dropping Dimensions and Measures.
‚óè‚Äã REQ-3.2: The system shall automatically generate the necessary backend code (SQL) to
fetch data, so the user never has to write code manually.
‚óè‚Äã REQ-3.3: The system shall allow users to apply Filters (e.g., equals, greater than,
contains) to narrow down the data being queried.
‚óè‚Äã REQ-3.4: The system shall provide a Live Preview table that shows the first 10 rows of the
query results so the user can verify their data before moving to visualization.
‚óè‚Äã REQ-3.5: The system shall allow users to Save Queries, so they can use the same data
selection for multiple different charts later.
‚óè‚Äã REQ-3.6: The system shall prevent "Invalid Queries" by only allowing users to perform
math (Measures) on numerical data and grouping on categorical data (Dimensions).

3.8 Alerts system
3.8.1 Description and Priority
Priority: Medium.
The system features an automated alerting system that monitors data for specific conditions. While

3.8.2 Stimulus/Response Sequences
‚óè‚Äã A data measure checks pre-defined conditions, such as a sharp drop in sales.
‚óè‚Äã The system triggers an alert event and sends the relevant data to the notification team‚Äôs

‚óè‚Äã The external notification team successfully delivers the message.
‚óè‚Äã The system receives and displays a "notification of success" from the alerting system for

3.8.3 Functional Requirements
‚óè‚Äã REQ-8.1: The system shall allow users to set custom "Alert Rules" based on specific

‚óè‚Äã REQ-8.2: The system shall automatically monitor data streams and batch uploads to detect
when these rules are violated.
‚óè‚Äã REQ-8.3: The system shall generate an alert payload (including chart data and text) and
pass it to the external notification team‚Äôs API.
‚óè‚Äã REQ-8.4: The system shall listen for and record a "success" confirmation from the external
notification system once a message is sent.
‚óè‚Äã REQ-8.5: The system shall provide an internal log for users to track which alerts were
triggered and whether they were successfully dispatched.
